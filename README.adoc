= The Compiler as Oracle — arXiv Preprint
:description: LaTeX source for the kUML research paper "The Compiler as Oracle: Reliable LLM-Based Architecture Modeling via a Typed Internal DSL", targeting arXiv (cs.SE).

image:https://zenodo.org/badge/DOI/10.5281/zenodo.21327229.svg[DOI,link=https://doi.org/10.5281/zenodo.21327229]

*Citable now via Zenodo:* https://doi.org/10.5281/zenodo.21327229[10.5281/zenodo.21327229] (v1.1.0, 2026-07-12 —
adds three figures and the Typed Domain Grounding terminology; still the single-model result set).
Earlier version: https://doi.org/10.5281/zenodo.21219554[10.5281/zenodo.21219554] (v1.0.0, 2026-07-06).
v1.1.0 is a separate Zenodo record (own concept DOI, `10.5281/zenodo.21219553`) rather than a new version
of v1.0.0, because v1.0.0 was uploaded manually before the GitHub-Zenodo integration existed — see
`.zenodo.json`'s `related_identifiers` for the cross-link. The arXiv submission (cs.SE) is pending
endorsement — see https://github.com/kuml-dev/kUML[the main kUML repository] and
https://kuml.dev/blog/2026-07-06-gcr-benchmark-kuml-vs-plantuml-mermaid[the benchmark write-up]
for context in the meantime.

[.lead]
*Reliable LLM-Based Architecture Modeling via a Typed Internal DSL.* +
Argues that a statically typed internal DSL (kUML, embedded in Kotlin) turns the host
compiler into a free, deterministic validation oracle for LLM-generated architecture
models — closing a Generate-Compile-Repair (GCR) loop without a second model, grammar-
constrained decoding, or fine-tuning. Reports a 50-task benchmark comparing kUML, PlantUML,
and Mermaid.

== Vorlage

`\documentclass[acmsmall,nonacm]{acmart}` — einspaltiges Preprint-Layout ohne
ACM-Rechte-Block, passend für arXiv vor einer Venue-Einreichung. Alle drei
Zielkonferenzen (ICSE, ASE, SLE) nutzen `acmart`; für eine Venue-Einreichung genügt
ein Wechsel der Klassenoption (z. B. `sigconf` für ein zweispaltiges Konferenz-Layout)
ohne Änderungen am Fließtext.

== arXiv-Einreichung — Checkliste

* `make pdf` lokal sauber durchlaufen lassen (keine Overfull-Boxen, keine fehlenden Zitate).
* Für die arXiv-Quellen-Einreichung zusätzlich das generierte `paper.bbl` mit hochladen
  (nicht nur `references.bib`) — vermeidet Probleme, falls arXivs TeX-Live-Snapshot
  `ACM-Reference-Format.bst` einmal nicht mitbringt. `paper.bbl` entsteht durch einen
  vollen `latexmk`-Lauf im `build/`-Verzeichnis.
* Kategorie: cs.SE (Software Engineering).
* Lizenz/Rechte-Feld auf arXiv: kein ACM-Copyright (die Vorlage ist bereits mit
  `nonacm` und `\setcopyright{none}` konfiguriert).

== Neue Zenodo-Version veröffentlichen

[NOTE]
====
**Erledigt für v1.1.0** (2026-07-12): Integration ist aktiv, Release ist raus, Zenodo hat
den Eintrag erzeugt — https://doi.org/10.5281/zenodo.21327229[10.5281/zenodo.21327229].
Die folgende Anleitung gilt jetzt nur noch für **künftige** Releases (v1.2.0 etc.), bei
denen einfach Schritt 3 (Tag + Release) reicht — Schritte 1–2 (Autorisierung) sind bereits
erledigt.
====

v1.0.0 wurde manuell hochgeladen, ohne GitHub-Zenodo-Integration — ein Release über die
Integration legt daher einen **eigenständigen, neuen Zenodo-Eintrag** an (eigene
Concept-DOI `10.5281/zenodo.21219553`, statt einer neuen Version von v1.0.0s Concept-DOI).
`.zenodo.json` verlinkt beide Einträge über `related_identifiers` (`isNewVersionOf`)
füreinander, damit sie auffindbar bleiben. Ab v1.1.0 hängen sich alle weiteren Tags/Releases
automatisch als neue Version an *diesen* Eintrag (`10.5281/zenodo.21219553`) an — die
Eigenständigkeits-Problematik betraf nur den allerersten Übergang.

*Ablauf für einen neuen Release* (Autorisierung bei zenodo.org/account/settings/github/
ist bereits erledigt, muss nicht wiederholt werden):

[source,bash]
----
git tag -a vX.Y.Z -m "Kurzbeschreibung der Änderungen"
git push origin vX.Y.Z
gh release create vX.Y.Z --title "vX.Y.Z — Kurztitel" --notes "Ausführliche Release-Notes"
----

Zenodo greift den Release automatisch auf (üblicherweise innerhalb weniger Minuten).
Danach die neue DOI in `README.adoc` (Badge + Zitierhinweis oben) und `CITATION.cff`
nachtragen — wie bei v1.1.0 hier bereits geschehen.
